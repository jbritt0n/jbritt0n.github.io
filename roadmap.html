S.rotationSpeed — Implementation Notes

Is it currently in the code? Partially. S.charRotation exists as a static rotation angle (a fixed offset applied to all characters), but there is no variable controlling the rate at which characters rotate over time. Rotation speed is currently hardcoded per-mode as a multiplier on t — e.g. Bloom uses t * 0.5, Vortex uses (maxR / r) * 0.5, Tessellation's pulse uses t * 2. These are different in kind: some control angular velocity of path motion, some control how fast a noise-derived angle changes.

ModeCurrent hardcoded rotation rateWhat S.rotationSpeed multiplies
Bloomt * 0.5 in angle noiseReplace 0.5 with 0.5 * S.rotationSpeed. Controls how fast the bloom angles drift.
Vortex(maxR/r) * 0.5 angular velocityReplace inner 0.5. Controls spin rate of the whole vortex.
Tessellationt * 2 in pulse sinReplace 2. Controls how fast cell phases oscillate.
Lissajouspt.speed * (S.speed*50 + 0.5)Add * S.rotationSpeed to phase increment. Controls parametric curve traversal speed.
Spiralt * 2 in wobbleReplace 2. Controls how fast the spiral wobble oscillates.
FractalSubdivImplicit via t in depth animationsMultiply t component by S.rotationSpeed before use.
OrbitAngle advances each frame by fixed stepAdd * S.rotationSpeed to the per-frame angle increment.
WireTerraint * S.speed * 40 camera orbitAlready partially controlled by S.speed. S.rotationSpeed would be a dedicated rotation-only multiplier separate from global speed.
Recommended implementation: Rather than one global S.rotationSpeed that touches all modes (which risks making unrelated modes feel the same), implement it in two tiers:
1. Global S.rotationSpeed (default 1.0) applied as effectiveT_rot = t * S.rotationSpeed used only in angle calculations. Keeps motion speed and rotation speed independent.
2. Per-mode rotMult sliders (bloomRotMult already planned) that scale on top of the global value. So Bloom's final angle speed = t * S.rotationSpeed * bloomRotMult.
At S.rotationSpeed = 0: all rotation freezes but all other motion (particle movement, size pulsing) continues — this is a creatively useful state.

Augmented Slider UX

Sliders gain a modulation source attachment point. Keep casual UX identical — the modulation icon is a subtle dot that expands on hover. Advanced users discover it; beginners never see it.

{
  value: number,       // current base value
  min: number,
  max: number,
  modSource: null | {
    type: 'sine' | 'noise' | 'random',
    freq: number,      // LFO frequency
    amp: number,       // modulation depth (fraction of range)
    phase: number      // phase offset
  }
}

// evaluated value at time t:
effective = value + (modSource ? evaluateMod(modSource, t) * (max-min) : 0)

Target Pipeline Model

All modes should gradually conform to this five-stage pipeline. Today, logic from all five stages is interleaved inside individual drawXXX() functions. The goal is progressive separation — not a big-bang rewrite.

Field — F(x,y,t)
Defines values at every point in space and time: noise density, velocity, color weight. Abstracted by sampleNoise() and the field registry.
Transform
Warps or modifies coordinates before sampling. Domain warping, jitter, Poincaré disk, gravitational lensing. Applied globally before any mode evaluates.
Sample
The structure used to query the field. Currently: grid (most modes), particles (Loops, Sand, Boids), curves (Calligraphy, Lissajous), image pixels (ImgText).
Render
How sampled field values become visuals. Text glyphs (most modes), shapes (overlay), strokes (Calligraphy, Seisma, TopoLines). Size, rotation, alpha from field values.
Post
Canvas-level effects applied after rendering. Pixel sorting, CRT, RGB flicker, glitch overlays. Mode-agnostic. Added in Phase C.

Analysis & Strategic Advice

Assessment of the architecture brief from a second model. The brief is well-structured and directionally correct. The following notes identify risks, sequencing issues, and where to push harder vs. pull back.

Strong — Do This
Noise abstraction layer is the right first move
The brief correctly identifies this as foundational. A single sampleNoise(x,y,t,type) function costs 1–2 hours and unlocks fBm, Voronoi noise, and curl noise without touching a single draw function. This should be the very first architectural change. The pre-computed shared grid is the bonus: do it at the same time, get 20–40% frame rate improvement for free.
Strong — Do This
Domain warping has the best creative-to-effort ratio of anything in the brief
Two lines of code per mode (wx = x + noise(...)*amp, wy = y + ...) plus two global sliders turns every single existing mode into something dramatically richer. This should be implemented as a pre-Transform step before field evaluation — all 38 modes benefit instantly without individual refactors. After noise abstraction, this is the second highest-priority change.
Strong — Do This
Post-processing is a free architectural win
Pixel sorting, CRT effects, and RGB flicker operate on the canvas pixel buffer after all drawing is complete. They require zero changes to any draw function and can be toggled on/off without affecting mode behavior. The modular glitch refactor (breaking Glitch mode into discrete POST operations) is the cleanest win here — it both simplifies Glitch and makes those effects available everywhere.
Strong — Do This
Augmented sliders are the right UX direction — scope them carefully
Adding a modulation source to sliders is exactly right. The key constraint: implement modulation attachment on the component level, not as a global system change. Build one AugSlider component, wire it to three or four high-value sliders first (density, wave, maxSize, speed), prove the UX, then roll out. Don't change all 60+ sliders at once.
Caution — Sequence Matters
The Bloom refactor is correct — but "template" means slow, not fast
Refactoring Bloom as the first pipeline-compliant mode is smart. The risk: if you refactor Bloom too quickly and discover the pipeline abstraction has edge cases, you'll propagate those edge cases to every subsequent mode. Spend extra time getting Bloom right. Let it sit in production for a week before using it as the template for mode two. A well-validated template saves 10× the time across 38 modes.
Caution — Sequence Matters
Full pipeline adoption must be incremental, not a flag-day rewrite
The brief describes the FIELD→TRANSFORM→SAMPLE→RENDER→POST model as the target, but doesn't specify the migration path. Critically: do not try to conform all 38 modes to the pipeline simultaneously. The right approach is a "wrapper shim" — the pipeline stages call existing draw functions initially, and individual draw functions get progressively refactored behind the shim. This keeps the studio functional throughout the refactor.
Caution — Sequence Matters
The layer system should come last, not be designed first
The brief positions the layer system (Phase 9 in its numbering) relatively early conceptually. In practice, the offscreen canvas architecture required for proper layering is non-trivial and has performance implications at high resolutions. The existing overlay/shape/composition block system already handles 80% of layering use cases. Implement RD-as-mask (a specific, bounded layer use case) first, validate the architecture, then build the general layer stack.
Risk — Watch This
The modulation system needs a "complexity budget"
Augmented sliders with per-slider LFO/noise/random modulation sources are powerful but create exponential UI complexity: 60 sliders × 4 modulation types × 3 parameters each = 720 invisible configuration points. Without discipline, a preset can become impossible to understand or reproduce. Recommendation: limit modulation to 8–10 "primary" sliders only (density, wave, speed, maxSize, a few mode-specific ones). Do not expose modulation on every slider.
Risk — Watch This
Noise type switching can break existing presets
When you add S.noiseType and change the default from Perlin to anything else, every existing preset that doesn't specify a noise type will behave differently. Mitigation: set the default to 'perlin' and make it explicit in every preset. Add a "legacy mode" flag that forces Perlin regardless of noiseType for backward compatibility. Test all presets after Phase A.
Risk — Watch This
Pixel sorting is expensive at high resolution
getImageData() / putImageData() on a full HD canvas at 60fps is a significant memory bandwidth hit. Pixel sorting sorts pixel runs — at full canvas size this can drop frame rate by 30–50% depending on threshold. Mitigation: run pixel sorting every N frames (3–5), not every frame. Add a quality slider that controls sort resolution (e.g., sort at 50% scale, upscale result). Make the performance cost visible in the UI.
Suggestion — Add This
The brief omits "global time control" — it should be earlier
S.timeScale as a global time multiplier is listed as Phase 1 in the brief but isn't in the critical path. This is wrong: it's one of the easiest global parameters to implement (multiply t once in the main loop) and one of the most expressive for users — slow motion, fast forward, pause-and-step. Do it in Phase 0 alongside the other trivial wiring tasks.
Suggestion — Add This
The brief doesn't address the F(x,y,t) → Color pipeline
The architecture describes F(x,y,t) driving density, motion, and size, but not color. Currently each mode handles color independently with different logic. A unified color evaluation step — where the field value maps to a position along the current palette — would be a massive consistency win. This is the "field → color" variant of the existing getColorIndex() function. Worth adding to Phase D explicitly.
Suggestion — Add This
RD simulation as a density mask should ship before the full layer system
The brief correctly identifies Reaction-Diffusion as an overlay use case but buries it in Phase 7. This specific feature — using the RD B-channel as a draw/skip mask for any mode — is: (a) implementable in ~30 lines, (b) requires no offscreen canvas, (c) produces extraordinary visuals (Bloom shaped by Turing patterns). Promote this to Phase F with a specific early-ship target, decoupled from the general layer system.
